# 买车票系统设计文档

## 1. 系统定位

买车票系统是一个面向多种出行方式的统一票务与订单平台，支持乘客查询班次、购票、支付、查看订单，也支持司机或车主发布行程、管理座位与收益，同时提供管理员后台做风控、监控、模型配置和权限管理。

系统目标：

- 支持汽车、火车、飞机、高铁、公交等多种票务场景
- 支持乘客端、司机端、管理后台三类核心用户
- 支持 AI 查询、智能推荐、知识库问答、自动创建订单等能力
- 支持高并发、低延迟、可监控、可扩展的系统架构

## 2. 出行类型

当前支持的车票或出行票务类型：

- 汽车
- 火车
- 飞机
- 高铁
- 公交

后续可以继续扩展：

- 地铁联程
- 城际拼车
- 旅游大巴
- 定制包车

## 3. 用户角色与权限

### 3.1 乘客

乘客是系统中的购票用户，核心能力包括：

- 注册、登录、实名认证
- 查询出发地、目的地、时间范围内的票务信息
- 查看班次、价格、剩余票数、座位类型、途经站点、预计到达时间
- 下单购票、取消订单、支付、退款
- 查看历史订单、电子票、乘车凭证
- 收藏常用路线
- 接收行程提醒、支付提醒、改签或停运通知
- 使用 AI 助手查询最优路线、最省钱方案、最快到达方案

### 3.2 司机 / 车主 / 运力发布方

司机端负责发布和维护可售班次，核心能力包括：

- 发布车次或行程信息
- 配置出发时间、起点、终点、途经站点
- 配置每个站点的大致到达时间
- 配置票价、票数、座位类型
- 查看订单情况和乘客信息
- 处理接单、核销、出发、完成行程
- 查看总收益、日收益、月收益、订单数、上座率
- 接收 AI 辅助，快速生成标准化行程信息或订单草稿

### 3.3 管理员

管理员负责后台运营、风控和系统治理，核心能力包括：

- 管理后台订单
- 管理用户信息、状态、角色、权限
- 管理各个用户的 token 使用情况
- 管理各个用户的 token 限制
- 查看 API 使用风险、风险告警
- 查看平台运营监控数据
- 配置 AI 模型与调用策略
- 管理知识库内容和 RAG 检索策略
- 管理其他用户权限
- 控制 MCP 工具的启用、停用和授权范围

## 5. 核心业务流程

### 5.1 乘客购票流程

1. 乘客输入起点、终点、日期
2. 系统返回可选班次列表
3. 乘客查看票价、时间、余票、经停站点
4. 乘客选择班次和座位类型
5. 系统创建待支付订单
6. 用户完成支付
7. 系统锁定座位并出票
8. 乘客在订单页查看电子票和乘车信息

### 5.2 司机发布班次流程

1. 司机登录司机端
2. 填写出发时间、起点、终点、途经站点、预计到达时间
3. 填写票价、总座位数、可售票数
4. 提交审核或直接发布
5. 系统生成班次
6. 乘客可在前台查询到该班次

### 5.3 管理员风控流程

1. 管理员查看 API 调用日志与告警面板
2. 系统识别异常 token 消耗、异常访问频率或风险请求
3. 管理员调整限流策略、token 配额、模型权限
4. 必要时冻结用户、封禁接口或停用 MCP 工具

## 8. AI 与 RAG 系统设计

### 8.1 AI 能力定位

系统内部署自己的 RAG 知识库与大模型，主要用于：

- 帮乘客查询车票信息
- 帮用户选择最优、最省钱、最快捷的出行方案
- 帮司机快速创建订单或班次
- 提供智能客服问答
- 帮管理员识别异常调用和潜在风险

### 8.2 知识库内容

RAG 知识库可接入以下内容：

- 车票规则文档
- 退票与改签规则
- 平台使用说明
- 常见问题 FAQ
- 司机发布规范
- 风控规则说明
- API 文档和后台操作手册

### 8.3 AI 查询流程

1. 用户输入自然语言问题
2. 系统做意图识别
3. 从知识库检索相关文档片段
4. 将检索结果和用户问题一起发给大模型
5. 大模型生成回答
6. 系统做安全审查、敏感词过滤和结果兜底
7. 返回给用户

### 8.4 自然语言处理能力

- 识别起点、终点、日期、人数、预算
- 理解“最便宜”“最快”“今天晚上”“明天上午”等自然语言表达
- 将用户意图转换成结构化查询参数
- 自动补全缺失信息
- 对不明确的问题进行追问或推荐

### 8.5 AI 生成能力

- 为司机生成标准化行程描述
- 为管理员生成风险摘要
- 为乘客生成出行建议
- 自动生成客服回复草稿

## 9. 大模型部署设计

### 9.1 模型部署方式

可以采用以下部署方式：

- 本地 GPU 部署开源模型
- 私有云部署推理服务
- 企业内部网关统一转发模型请求

### 9.2 模型分层

- 小模型负责意图分类、实体抽取、风控初筛
- 大模型负责复杂问答、总结、推荐、生成
- RAG 检索层负责召回知识库内容

### 9.3 模型路由策略

- 简单查询优先走轻量模型，降低成本
- 复杂问答走大模型，提高效果
- 高风险请求先过审查模型
- 超长上下文问题走 RAG + 长上下文模型

## 10. AI 优化方向

系统需要重点解决以下问题：

- 高并发下的稳定响应
- 低延迟的查询体验
- 脏数据、重复数据、缺失数据处理
- 复杂检索和多条件召回优化
- 大模型幻觉拦截
- GPU 算力利用率优化

建议优化点：

1. 建立自动化评估流水线，例如接入 RAGAS 等评测工具
2. 建立全链路监控和可观测性体系，监控检索命中率、回答质量、响应耗时、token 消耗、错误率
3. 建立回答质量评分和告警机制，发现质量下降时可以快速定位问题
4. 引入缓存、召回排序、重排模型，减少重复推理成本
5. 建立敏感问题拦截、风险输出审核和人工兜底机制

## 11. 系统技术架构

### 11.1 架构分层

- 前端层：Web 端、移动端、管理后台
- 网关层：统一鉴权、限流、日志、路由转发
- 业务层：用户服务、票务服务、订单服务、支付服务、司机服务、管理后台服务
- AI 层：意图识别、检索服务、RAG 服务、大模型推理服务
- 数据层：MySQL、Redis、向量数据库、对象存储、日志系统

### 11.2 推荐技术组件

- 前端：Vue 或 React
- 后端：Java Spring Boot / Node.js / Python FastAPI
- 关系数据库：MySQL 或 PostgreSQL
- 缓存：Redis
- 消息队列：RabbitMQ 或 Kafka
- 搜索与检索：Elasticsearch + 向量数据库
- 文件存储：MinIO 或对象存储服务
- 监控：Prometheus + Grafana
