# TripVerse 前后端分离规范

## 1. 当前项目结构

当前仓库已经按“前后端同仓开发”整理为以下结构：

```text
RideHailingSystem/
  backend/
    src/
  docs/
    买车票.md
    前后端分离规范.md
    前端接口文档.md
  frontend/
    assets/
    admin-*.html
    driver-*.html
    *.html
  .editorconfig
  README.md
```

目录职责：

- `frontend/`：前端原型、静态资源、运行时配置
- `backend/`：后端代码目录
- `docs/`：需求、设计、规范、接口文档

## 2. 当前前端规范入口

### 2.1 运行时配置

文件：`frontend/assets/runtime-config.js`

用途：

- 统一前端运行配置
- 开发、测试、生产环境切换时统一改这里
- 后续接后端接口时，主要修改 `apiBaseUrl`

当前配置字段：

- `appName`
- `apiBaseUrl`
- `timeout`
- `useMock`
- `debug`

建议配置：

- 本地开发：`http://localhost:8080/api`
- 测试环境：`https://test-api.xxx.com/api`
- 生产环境：`https://api.xxx.com/api`

### 2.2 公共脚本

文件：`frontend/assets/app.js`

当前负责：

1. 前端运行时初始化
2. 路由常量管理
3. API 路径常量管理
4. 通用请求封装
5. 页面公共交互初始化

当前暴露的全局对象：

- `window.TripVerse.config`
- `window.TripVerse.routes`
- `window.TripVerse.endpoints`
- `window.TripVerse.api`

示例：

```js
TripVerse.api.get(TripVerse.endpoints.passenger.myOrders)
TripVerse.api.post(TripVerse.endpoints.passenger.orderPreview, payload)
```

## 3. 推荐的正式前端结构

当前 `frontend/` 是静态原型目录。后续做正式前后端分离时，建议逐步演进为：

```text
frontend/
  public/
    runtime-config.js
  src/
    api/
      http.js
      passenger.js
      driver.js
      admin.js
    constants/
      routes.js
      endpoints.js
    components/
      common/
      layout/
      business/
    pages/
      passenger/
      driver/
      admin/
    styles/
      global.css
      variables.css
    utils/
    main.js
```

## 4. 推荐的后端结构

后端建议只在 `backend/` 目录内初始化，不要再把后端文件放回项目根目录。

一个通用后端结构示例：

```text
backend/
  src/
    controllers/
    services/
    repositories/
    models/
    middlewares/
    config/
    routes/
  tests/
  scripts/
  README.md
```

如果你使用 Java，也可以是：

```text
backend/
  src/main/java/
  src/main/resources/
  src/test/java/
```

## 5. 页面与接口分层建议

后续不要在页面里直接写死接口地址，建议拆成三层：

### 5.1 接口路径常量层

```js
export const PASSENGER_API = {
  searchTickets: "/tickets/search",
  myOrders: "/orders/my",
}
```

### 5.2 请求层

```js
export function get(url, params) {}
export function post(url, data) {}
export function put(url, data) {}
export function del(url) {}
```

### 5.3 业务服务层

```js
export function fetchMyOrders() {
  return get(PASSENGER_API.myOrders)
}
```

这样做的好处：

- 页面不直接依赖底层请求库
- 后续从 `fetch` 切换到 `axios` 成本很低
- 后端网关路径变化时只需要改一层

## 6. 命名规范

### 6.1 目录命名

- 目录统一小写
- 多单词目录优先使用短横线或语义清晰的单词

### 6.2 文件命名

静态页面阶段：

- `driver-dashboard.html`
- `admin-orders.html`
- `order-detail.html`

组件化阶段：

- `DriverDashboardPage.vue`
- `AdminOrdersPage.vue`
- `OrderDetailPage.jsx`

### 6.3 代码命名

- 常量：`UPPER_SNAKE_CASE`
- 函数：`camelCase`
- 类 / 组件：`PascalCase`
- 接口模块：按业务命名，例如 `passengerApi.js`

## 7. 文档放置规范

以后文档统一放在 `docs/`，建议至少区分这几类：

- 需求文档
- 系统设计文档
- 接口文档
- 数据库设计文档
- 部署文档

建议命名示例：

- `docs/系统设计.md`
- `docs/接口文档.md`
- `docs/数据库设计.md`

当前已经补充：

- `docs/买车票.md`
- `docs/前后端分离规范.md`
- `docs/前端接口文档.md`

## 8. 当前阶段最推荐的推进顺序

1. 在 `backend/` 初始化正式后端项目
2. 先定义接口文档，放在 `docs/`
3. 把 `frontend/assets/runtime-config.js` 的 `apiBaseUrl` 改成后端地址
4. 优先联通乘客端主链路：搜索、详情、下单、支付、订单
5. 再联通司机端和管理后台
